1. 设计原则

核心结论

五条原则，从痛点到原则

每条原则对应一个曾经踩过的坑。

① 唯一真相源
每条数据只有一个家，所有视图都是读出来的。真相在文件里，不在服务里。服务是代理，不是主人。
② 骨肉物理隔离
骨件（C 盘）可复制，肉件（D 盘）是成果。骨头负责运行，肉负责产出。骨挂了不影响项目运行。
③ 薄工具架
定义在项目，执行在调度器，面板做可见性。工具架熄了，项目照跑。面板是窗户，不是发动机。
④ 独立可替换
每块骨头是独立进程+独立目录，通过文件读写通信。一块挂了不影响其他。可单独 git，可单独重启，可单独替换。
⑤ 设计以痛点为锚
不预建框架等坑来填。每个功能对应一个曾经真实发生的失败。没被坑过的不建独立骨件。

三大当前痛点

#痛点根因对应设计
1备份不完整，恢复路径不确定备份分散，无统一验证warehouse 统一备份 + 各骨 git push
2服务状态不可见端口/进程/产出无统一视图巡检 cron job + 漂移检测 + loop monitor
3AI agent 行为无边界规则散落，无硬阻断CLAUDE.md 宪法 + 权限弹窗 + 红线

核心结论

四个可验证目标 + 四个明确不做

成功标准（可验证）

#目标验证方式

G2项目独立运行，不依赖面板关掉 agentboard :3099，cron 任务照跑

#不做的事为什么

N2审批流引擎（guardrails 骨）宪法红线 + 权限弹窗 + 人肉确认已够用。等红线痛点出现再自动化
N3bootstrap 骨件派生品不配当骨。各骨件自己的 setup 自己管（各自 git repo 自带）
N4自动自愈先做到可见，再做自愈。盲人摸象式自愈比不治更危险

3. 方案选型

核心结论

薄工具架 4/4 胜出

厚工具架 vs 薄工具架。用 Karpathy 四标准判胜负。

维度厚工具架（框架式）薄工具架（联邦式）
定义在哪面板是主人，工具向面板注册定义在项目，面板只读
执行在哪面板调工具调度器独立进程，面板不参与执行路径
面板挂了所有工具不可见、不可管项目照跑，只失去可见性
加新项目改面板代码只加 manifest.json

Karpathy 四标准判胜负

标准厚工具架薄工具架
① 关键路径最短面板在关键路径面板不在关键路径
② 故障隔离最好面板 = 单点骨肉天然隔离
③ 新人最快理解需理解框架约定看 manifest.json 即懂
④ 改动最少牵连改面板影响全局改一个项目只动一个 manifest

结论：薄工具架 4/4 全胜。唯一真相源是项目目录里的文件，不是面板服务。

4. 详细设计

核心结论

三块骨头 + 八组件健康

骨头负责运行，八组件负责每个项目的健康度量。骨可复制，肉是每个人自己的成果。

4.1 三骨件

骨 = 三块：合约 → 目录 → 程序
骨的本质是可复制的设计模式。三块骨头覆盖 AI Agent 运行所需的全部基础设施：

① ~/.claude/ — Memory & Knowledge + Guardrails。宪法 + 技能 + 记忆 + 红线规则。AI 的行为边界由这里定义。
② ~/.agentboard/ — Protocols & Tools。工具架 + 面板 + manifest 注册。工具的发现、启停、监控界面。
③ ~/.scheduler/ — Execution + Observability。调度器 + 执行器 + 运行时状态 + 巡检。独立进程 :3100。

为什么没有 eval / guardrails / bootstrap？
· eval — 操作日志 + cron 状态 + 漂移检测已覆盖当前巡检需求。等被 agent 幻觉骗了再建二元断言模版，不独立成骨。
· guardrails — 宪法已有红线，权限弹窗 + 人肉确认已够用。等红线痛点出现再自动化。
· bootstrap — 派生品，无自有真相源。各骨件自己的 setup 自己管（各自 git repo 自带 README）。
骨 — 基件，运行在 C 盘 肉 — 肌件，属于各项目（D 盘） 紫 — 接口/协议

骨件目录结构

~/.claude/ ├── CLAUDE.md # 宪法（全局行为规则） ├── skills/ # 技能定义 ├── settings.json # Claude Code 配置（MCP / Hooks） └── projects/.../memory/ # 跨会话记忆 ~/.agentboard/ ├── server.js # HTTP :3099 — 可见性层 ├── mcp-server.js # MCP stdio — AI 接口 ├── index.html # 工具架面板 ├── lib/ │ ├── tool-registry.js # 工具注册 + 扫描 │ └── manifest-schema.js # Manifest 标准唯一真相源 ├── tools/ # 各工具注册目录（42 工具） ├── tips/ # 操作日志 └── _runtime/ # 临时文件 ~/.scheduler/ ├── scheduler.js # 调度器（独立进程 :3100） ├── runner.js # 任务执行器 ├── runtime-logger.js # 运行日志写入 ├── scheduler-state.json # 运行时状态（调度器写，面板读） ├── crontab.txt # 全局定时定义 └── task-schema.json # 任务定义标准 │ │ 调度器是独立进程 + 独立目录。不寄生在面板下。 │ 系统调度守护 → 读 crontab.txt → 扫描项目 → 调 executor → 写 state.json。 │ 面板 server 只读 scheduler-state.json 做面板展示——不在执行路径上。

4.2 八组件健康模型

每个项目自评 8 个维度，写入 health.json。面板只读。
#组件检查什么归于哪个骨
1闹钟定时任务有没有按时触发③ scheduler
2工位执行模式：串行/并行，有无冲突③ scheduler
3知识管理CLAUDE.md 存在 + memory + 踩坑警示① claude
4电话线外部 API / 服务连接状态② agentboard
5质检员产出是否经过独立验证（防自己判自己满分）③ scheduler (巡检)
6记事本状态文件 + 事件日志是否写入③ scheduler
7提报失败通知通道（飞书 / Slack / 邮件）③ scheduler
8目标定义完成标准是否可验证（非自然语言）① claude

8 组件归入 3 骨，闭环。质检员 (inspector) 替代了独立 eval 骨的功能——操作日志 + cron 状态 + 漂移检测 = 当前的质检实现。

接口格式谁写谁读

唯一硬接口：health.json
任何工具通过这个文件读项目状态。不调 API，不连数据库，不依赖面板。文件在项目在。

4.4 备份体系

层级内容方式
骨件级~/.claude/ ~/.agentboard/ ~/.scheduler/各自 git repo，独立 push
项目级workspace 下各项目各自 git repo，仓库清单见 repo-spec.md
产出级第三方工具源码D:\tools\ 下各自 git
统一仓库F:\warehouse\inbox\备份落盘 + 恢复验证
灾难恢复
新机 → git clone 三骨 → git clone 项目 → 各骨 README/setup → 跑起来。不建 bootstrap 骨——各骨的 setup 各管各的，不派生第四个目录来复述前三者。

核心结论

止血 → 补基 → 标准化 → 自愈

四阶段演进。不是瀑布——前一阶段完成再进下一阶段。

施工纪律

  1. 每项任务改前备份
  2. 改完立即验证，不攒到 Phase 结束时验
  3. 验证不过不进下一项
  4. 触及承重墙 → 停，先讨论
  5. 每个 Phase 结束后更新 health.json score

核心结论

#决策理由

D-02薄工具架胜出Karpathy 四标准 4/4 全胜。定义在项目，执行独立，面板做可见性
D-03五组件归属各归其位安全 + 解耦 + 并行安全
D-04唯一硬接口：notebook/health.json任何工具通过文件读项目状态，不调 API
D-05架构文档按推导链重排参考 → 分析 → 目标 → 选型 → 设计 → 施工 → 决策
D-06文件管理规范从架构分离架构聚焦系统设计；文件管理升到 workspace 级
D-07方案选型只比两个真正可选的方案不拿集中式当靶子；不搞伪二选一
D-08产出物统一存放，不进项目生命周期产出物（文章/图片/视频）不是项目
D-09巡检规则独立设计覆盖 git / 备份 / 产出 / 日志 / 磁盘 / 交叉验证
D-10生命周期边界改为目录位置lab/ → 根目录 → _archive/，人决定搬移时机
D-11工具源码统一位置，与注册表分离注册表（~/.agentboard/tools/）和源码分离
