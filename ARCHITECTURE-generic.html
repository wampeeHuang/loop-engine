核心结论

七案例，七决策

分析了 4 个理论框架 + 3 个实战系统后形成的设计决策——不是"看到了什么"，是"决定了什么"。

纳入设计 · 7 项
  1. 反馈环思维——设计让 AI 去 prompt AI 的环，不做一次性指令（Osmani + Boris + Steinberger）
  2. 失败必须可见——所有 loop 有可观测输出，禁止静默失败（全部案例的共同底线）
  3. 文件通信、Agent 不互调——KUN ./state/*.json + OKF Markdown+YAML + Steinberger VISION.md
  4. 对错可判交机器，判断留给人——Boris Underfund + Koka L1→L2→L3 质检模型
  5. 先跑通最简单的，再复制模式——wulixfr 第一天 1 个 Agent + KUN 4 Agent 独立运行
  6. 成本护栏——KUN Studio 审批线，用约束驱动自动化而非堆资源
  7. git worktree 多项目隔离——多 Agent 并发操作不同项目时，各自在独立 worktree 里互不干扰
明确拒绝 · 6 项
  1. 数百并行实例（Boris）——单机按 token 付费，跑不起
  2. Linux cron + VPS（KUN）——Windows 原生调度器是既定环境
  3. Gemini 免费层（wulixfr）——模型策略不同
  4. 链式 Agent 路由（Steinberger）——单机过度工程，直接调比多层路由简单
  5. 纯文本质检（Koka）——需覆盖代码 + 数据 + 配置，维度更广
  6. 云服务托管（OKF）——本地文件系统 + git 完全够用
映射到后续设计

  2. Koka 三层 + Boris 对错可判 → inspector 质检组件（§4 详细设计）
  3. KUN 审批线 → 成本护栏（§4.4）

  5. wulixfr "第一天跑通" → 初始化向导（§5 分阶段施工）

0.1 理论基石

四位定义了 Loop Engineering 的理论框架。每位的贡献不同，批评也不同。调研方法：读原文 → 搜社区反馈 → 提取对我们的意义。

展开 4 个理论框架分析
Loop Engineering
Addy Osmani · 2026-06-07 · 原文

核心论点：人从"写 prompt 的人"变成"设计系统的人"。Loop = Automations + Worktrees + Skills + Connectors + Sub-agents + State/Memory。关键是设计反馈环，不是写一次性指令。

六原语全收：

Automations
→ 闹钟 · 定时触发
Worktrees
→ 工作区隔离 · 多项目并发保护
Skills
→ 工位 · 执行技能
Connectors
→ 电话线 · 外部连接
Sub-agents
→ 子代理 · 委派子任务
State/Memory
→ 记事本 · 状态记忆

两个关键警告：
comprehension debt（认知负债）——loop 跑得越快，人对代码的理解差距越大。
cognitive surrender（认知投降）——人放弃思考，loop 输出什么就接受什么。

社区批评

"旧酒新瓶"——Automations=cron(1975)、Worktrees=git worktree(2015)。Token 成本爆炸——loop 模式消耗 3-8× token。调试噩梦——47 轮 loop 失败后怎么定位？没有成熟的 loop 可观测性方案。

借鉴 反馈环设计思维。comprehension debt 概念直接纳入设计原则。cognitive surrender 提醒——人在 loop 中的角色是策展和判断，不是旁观。
不借鉴 Google Cloud ASR 案例——我们不需要那个复杂度。
术语：comprehension debt = loop 产出速度超出人类理解速度时积累的认知负债。 cognitive surrender = 人放弃主动判断、被动接受 AI 输出的状态。
Boris Cherny · WorkOS Unplugged
2026-06-02 · Claude Code 作者 · PMF 报道

核心论点："我不再直接 prompt AI 了。我写 loop，让 loop 去 prompt AI。"——同时跑数百个并行实例。Claude Code 现在 100% 由 Claude Code 自己写。

Underfund Principle（缺钱原则）：故意只给项目一半人手——人不够就会被迫自动化。"对错可判的任务交给机器，人只留在需要判断力的节点。"

社区反馈

两极分化：一部分人兴奋"这是 AI 编程的未来"，另一部分批评"97% 的人没有数百个并行实例的预算"。社区普遍认同：对错可判的任务才适合自动化。

借鉴 Underfund Principle——用约束驱动自动化而非堆人。对错可判交机器——inspector 质检组件直接继承这个思路。
不借鉴 数百并行实例——单机跑不起，token 账单受不了。
术语：Underfund Principle = 故意少给人手，用资源约束倒逼自动化。 对错可判 = 有明确可验证正确/错误标准的任务。
Peter Steinberger · Compound Operations
2026-06 · OpenClaw 作者 · @steipete · Digg 讨论

核心论点："你不应该再给 coding agent 写 prompt 了。你应该设计 loop，让 loop 去 prompt 你的 agent。"维护了每 5 分钟唤醒的 loop，部分全自动落地，高风险改动浮到人工 review。

Compound operations（复利操作）：prompt 是单次操作，loop 是复利操作——每次重复更便宜、更快、更好。

社区反馈

Token 成本——有人算了一笔账，5 分钟一个 loop 一个月烧几千刀。但被普遍认可的一点：loop 不应该静默失败。所有 loop 都有可见输出。

借鉴 Compound operations 复利思维——loop 不是"反复跑同一个东西"，是"每次比上次更好"。VISION.md 反馈机制。
不借鉴 链式 Supervisor→planner→coder→reviewer→tester 路由——单机过度工程。
术语：Compound operations = 每次循环复用并精炼上次结果的操作。 VISION.md = 项目目标声明文件，loop 用它自检是否偏离方向。
OKF · Open Knowledge Format
Google Cloud · 2026-06-12 · v0.1 · GitHub

核心论点：知识 = 带 YAML frontmatter 的 Markdown 文件目录——对人机同等友好。三原则：最小约束、生产者/消费者独立、格式不是平台（不需要 SDK、不需要 runtime、不需要账号）。

社区批评（Hacker News / Reddit）

"就这？Markdown + YAML 需要 15KB 规范？"但支持者认为：简单就是它的力量。Markdown+YAML 是人机互通的最大公约数。

借鉴 三原则——全部直接写入文件管理纪律。"1 个概念 = 1 个文件" → CLAUDE.md 分离策略。
不借鉴 Google Cloud 托管/推广流程——本地文件系统 + git 足够。

0.2 实战参考

三位在真实约束下跑了 6 个月以上的系统。他们的教训比理论家的建议更重。

展开 3 个实战系统分析
KUN Studio · 4 Agent SaaS
$15/月 Linux VPS · 130 项/天 · 6 个月稳定运行 · dev.to

核心架构：4 个 cron Agent 各司其职——Research、Writer、Editor、Publisher。Agent 之间不互相调用，通过 ./state/*.json 共享数据。$5 审批线：任何操作预估成本超 $5 必须人工确认。

借鉴 $5 审批线 → 计费审批线。Agent 不互调 → cron job 独立性设计。./state/*.json 共享状态 → health.json 就是这个模式。
不借鉴 Linux VPS + cron——我们是 Windows 原生调度器。
Koka Sexton · 三层质检机
16 项检查 · 管理 4 个内容资产 · 原文

L1 写前护栏：结构化内容简报在写作前强制执行——字数下限、设计元素数、AI 腔检测。
L2 编辑管线：Revise→Edit→Proofread 三阶段 + 16 点检查清单。
L3 周度扫查：4 个 cron job 每周跨资产扫查，每个错误反馈回 L1/L2 更新规则——闭环机制。

借鉴 L1→L2→L3 分层质检模型——直接映射到 inspector 组件设计。闭环反馈——inspector 查出的问题反馈到 alarm/ 规则更新。
不借鉴 纯文章质检维度→需覆盖代码、数据、调度、配置。
wulixfr · 9 Agent Python
~0€/月 · Windows schtasks · 免费层 · dev.to

核心架构：BaseAgent 五步骨架——load config → fetch → process → act → log。关键教训："先写骨架再造 Agent，第一天就跑通最简单那个。"

借鉴 BaseAgent 五步骨架。"第一天就跑通最简单的"——初始化向导的核心理念。Windows schtasks + 免费层——验证了可行性。
不借鉴 Gemini 免费层——模型策略不同。

1. 设计原则

从 7 个参考案例和工程实践中提炼。每条原则对应一个具体的失败模式——不是装饰，是教训。

核心结论

五条原则 · 八组件模型

五条原则从失败模式反推，八组件构成 Loop Engineering 的完整骨架。不是凭空设计——每条原则可以追溯到具体的失败案例。

五条设计原则
  1. 骨肉物理隔离——骨件（工具架/宪法/skills）和肉件（项目/数据）分盘存放，复制即迁移
  2. 失败必须可见——静默失败比报错危险。任何执行不论成败都有日志
  3. 唯一真相源——每条数据只有一个家。真相在文件里，不在服务里
  4. 薄工具架——定义在项目，执行在独立调度器，面板做可见性。面板熄了项目照跑
  5. 一次只改一个首要方向——不铺开所有改进同时做
八组件模型
🔔 闹钟定时触发，cron 调度
🏗 工位git worktree 并行隔离
📖 员工手册CLAUDE.md + handbook/
📞 电话线外部通道，token 隔离
🔍 质检员三层质检（写前→编辑→扫查）
📝 记事本health.json + events.jsonl
🚨 提报L0-L3 四级升级规则
🎯 目标定义战略/战术/操作三层目标
约束坐标系
  1. 单机——不引入分布式架构
  2. 零外部服务依赖——面板停了，项目照跑；API 挂了，数据还在
  3. 按 token 付费——不是内部无限额度，每次调用都计费
  4. 文件 > API——agent 之间用文件通信，不走 HTTP
  5. 工位即隔离机制——git worktree 实现安全并行
一句话总结

强大的 Agent 民兵单位——武器（工具注册）弹药（API 调用）阵地（项目目录）齐全，需要的是可见性 + 自检 + 告警三层指挥链。改造方向不是换武器，是建立指挥链。

1.1 Loop Engineering 通用痛点

每个接入 loop 体系的项目都会遇到的典型问题。

1
无声失败
任务报"成功"但实际空跑，无日志无从排查
2
多真相源
同一数据两处有，不知道信谁
3
备份无体系
口头原则，无脚本，无验证
4
无日志
问题发生了都不知道，事后无据可查
5
无安全护栏
误操作没有拦截，红线没有程序化
6
跨会话继续难
上下文断了，任务状态丢失
7
文件夹混乱
无统一目录约定，新项目东一块西一块
8
直接覆盖不迭代
改文件 = 全量覆盖，无 diff、无 git blame、无回滚
9
多调度器耦合
多套调度共存，职责重叠，单点故障风险

2. 设计目标

核心结论

八条验收标准 · 一个硬选择

八条标准是闸门不是愿望清单——每条有可验证的通过条件。一个硬选择：日志不论成败是前提，其他七条依赖它先成立。

展开验收标准详情（8 条成功标准 · 6 条 Non-Goals）
硬选择：失败可见优先

日志不可见 → 质检无从谈起 → 自愈不可信 → 独立运行是空话。先攻这一个方向，其余锦上添花。

五条设计原则（从失败反推）
  1. 骨肉物理隔离——复制即迁移
  2. 失败必须可见——静默比报错危险
  3. 唯一真相源——数据在文件，服务是代理
  4. 薄工具架——定义在项目，执行独立，面板做可见性
  5. 一次只改一个首要方向
产品化后再议 · 六条
  1. SaaS 多用户面板
  2. Prometheus/Grafana 实时监控
  3. Docker / K8s 容器化
  4. 通用 AI 自动修 bug
  5. 替换已有知识库
  6. 跨机器 agent 调度

2.1 成功标准（可验证）

#标准怎么验
G1kill 面板，项目独立跑完 cron 任务系统原生调度 + crontab.txt 兜底，手动跑一遍
G2任意项目复制到新机器，能直接工作项目自包含：CLAUDE.md + notebook/ + alarm/ 全在目录内
G3任何一次 cron 执行，不论成败，都有日志事件日志有记录，包括 stderr
G4任何一个文件操作失误，可以回滚git + 备份 + 回收站机制

3. 方案选型

每个决策只比较两个真正可选的方案。不拿烂方案当靶子。

核心结论

薄工具架 4/4 全胜 · 五组件各归其位

不是"哪个好"的偏好问题——是四个硬验证标准的直接比较。每个组件独立判断归属，不搞一刀切。

展开方案选型详情（厚 vs 薄 · 五组件归属细则）
Karpathy 四问定胜负
1. 杀面板，项目能跑？厚× 薄✓
2. 复制到新机直接工作？厚× 薄✓
3. token 安全？平手
4. 新项目接入改几处？厚 3+ 薄 0

厚工具架 1/4，薄工具架 4/4。不是偏好，是验证结果。

五组件归属（不是一刀切）
  1. 闹钟 → 定义在项目 alarm/，调度器独立执行，面板读状态
  2. 工位 → git worktree 隔离机制
  3. 员工手册 → 项目 CLAUDE.md + handbook/
  4. 电话线 → token 在面板，通道选择在项目 wire/
  5. 质检员+记事本 → 全在项目
选型方法：只比两个真正可选的方案

不拿烂方案当靶子，不搞伪二选一。厚 vs 薄是真正的架构分叉——两个方案各有真正优势。选薄不是因为厚是错的，是因为四标准测试厚过不了。

3.1 厚工具架 vs 薄工具架

这是整个架构的核心二选一。

变体 A · 厚工具架

定义 + 执行都在面板。项目只是被触发对象。

  • 闹钟定义 → 面板 cron/tasks.json
  • 员工手册 → 面板 skills/
  • 电话线 → 面板全管
  • 质检规则 → 面板 rules/
  • 记事本 → 面板 state/

真正优势：初期只要 1 个地方配

真正代价：每加一个项目面板胖一圈；项目无法脱离面板运行；换机器 = 手动重建面板配置

变体 B · 薄工具架

定义在项目，执行在独立调度器，面板做可见性。

  • 闹钟定义 → 项目 alarm/crontab.txt
  • 员工手册 → 项目 CLAUDE.md + handbook/
  • 电话线 → token 在面板，规则在项目 wire/
  • 质检规则 → 项目 inspector/checks.js
  • 记事本 → 项目 notebook/health.json

真正优势：项目自包含——复制文件夹即迁移；面板熄了项目照跑；配置随项目进 git；项目越多优势越明显

真正代价：前期多写几个文件；全局巡检需扫描多个文件

Karpathy 四标准判胜负

不靠感觉，靠验证
1. kill 掉面板，项目能不能独立跑完 cron？
  A：否（闹钟定义在面板，停了就没了）
  B：能（crontab.txt 在项目，系统调度兜底）
2. 项目复制到新机器能直接工作？
  A：否（需手动重建面板配置）
  B：能（项目自包含）
3. token 安全？
  平手——两变体 token 都在面板，不进项目仓库。
4. 新项目接入要改多少地方？
  A：面板改 3 处以上
  B：项目内新建 2 文件，面板零改动

A = 1/4，B = 4/4。变体 B 胜出。

3.2 五组件归属细则

组件本质决定理由
🔔 闹钟 定时触发 定义在项目 alarm/，调度器独立执行，面板读状态 调度需常驻进程；规则属于项目
🏗 工位 并行隔离机制 git worktree · 每项目独立 worktree 互不覆盖 git 原生能力，不绑定 Agent
📖 员工手册 项目知识 项目 CLAUDE.md + handbook/ 项目知识属于项目
📞 电话线 外部通道 token 在面板，通道选择在项目 wire/ 安全（token 不进仓库）+ 灵活（项目选通道）
🔍 质检员 独立验证 项目 inspector/checks.js，调度器执行，面板读结果 不同项目不同标准。Koka 三层质检模型复用
📝 记事本 跨会话状态 项目 notebook/health.json + events.jsonl "Agent forgets, the repo doesn't."

4. 详细设计

核心结论

骨肉分离 · 最小接口 · 项目自包含

六块骨头构成完整骨架，唯一硬接口 health.json，项目目录标准化。executor 可插拔——换 Agent 改一行配置，调度器无感。

展开详细设计详情（骨肉隔离 · 接口契约 · 项目骨架 · 备份体系）
已决设计 · 8 项
  1. 六骨 C 盘 · 肉件 D 盘——物理隔离，复制即迁移
  2. 唯一硬接口 health.json——status + score + tasks + meta 四个字段

4.1 骨肉物理隔离

骨 = 六块：合约 → 目录 → 程序
骨的本质是可复制的设计模式，不是特定的文件路径。六块骨头覆盖 O'Reilly 2026 AI Agents 六层栈中属于基础设施的五层：

① ~/.claude/ — Memory & Knowledge + Guardrails。宪法 + 技能 + 记忆 + 红线规则。
② ~/.agentboard/ — Protocols & Tools。工具架 + 面板 + manifest 注册。
③ ~/.scheduler/ — Execution。调度器 + 执行器合约 + 运行时状态。
④ ~/.eval/ — Eval & Observability（目标态）。质检注册标准 + 黄金测试集 + 漂移告警。
⑤ ~/.guardrails/ — Guardrails 执行层（目标态）。审批流引擎 + 红线规则执行器。
⑥ ~/.bootstrap/ — Bootstrap（目标态）。setup.ps1 + 恢复脚本。

④⑤⑥ 当前以合约形式存在于架构中，物理目录随施工生长。骨可复制，肉是每个人自己的成果。
骨 — 基件，注册在工具架 肉 — 肌件，属于各个项目 紫 — 接口/协议

工具架（骨）目录结构

~/.agentboard/ ├── server.js # HTTP 服务 + API — 可见性层 ├── apps-registry.json # 工具注册表（指针，指向工具目录） ├── loop-dashboard.html # 异常面板（标红不标绿） ├── tools/ # 各工具注册目录 ├── state/ # 工具架自身状态 └── _runtime/ # 临时文件 ~/.scheduler/ ├── scheduler.js # 调度器（独立进程，不依赖任何服务） ├── runner.js # 任务执行器 ├── runtime-logger.js # 运行日志写入 ├── scheduler-state.json # 运行时状态（调度器写，面板读） ├── crontab.txt # 全局定时定义（也支持项目级 alarm/crontab.txt） │ │ 调度器是独立进程 + 独立目录。不寄生在面板下。 │ 系统调度守护 → 读 crontab.txt → 扫描 workspace resolve 路径 → 调 executor → 写 state.json。 │ 面板 server 只读 scheduler-state.json 做面板展示——不在执行路径上。 ~/.claude/ ├── CLAUDE.md # 宪法（全局行为规则） ├── skills/ # 技能定义 └── projects/.../memory/ # 跨会话记忆 目标态骨目录（合约已定义，物理目录随施工生长）： ~/.eval\ # 质检注册标准 + 黄金测试集 + 漂移告警 ~/.guardrails\ # 审批流引擎 + 红线规则执行器 ~/.bootstrap\ # setup.ps1 + 恢复脚本

项目区（肉）目录结构

三段生命周期：lab → root → archive
目录位置是唯一分界线。lab/ = 孵化，根目录 = 稳定，_archive/ = 归档。移入移出由人决定。
项目名是唯一标识。所有指针（crontab、memory、文档引用）用项目名，不用路径。文件夹改名/移位后，调度器和面板下次扫描自动发现新位置。
<workspace>/ ├── CLAUDE.md # workspace 级规范（所有项目遵守） ├── _backups/ # 备份脚本 + 日志 │ ├── 项目名/ # 稳定期：有 git，去日期，纯项目名 │ ├── CLAUDE.md # 项目规则 │ ├── HANDOFF.md # 会话交接（gitignore） │ ├── notebook/ # 标准接口层 │ │ └── health.json # 健康数据 + projectName 字段（唯一 ID） │ ├── alarm/ # 闹钟定义 │ │ └── crontab.txt # cron expr | executor | project-name | task │ ├── inspector/ # 质检逻辑 │ │ ├── checks.js │ │ └── model-eval/ # 质量漂移检测 │ │ ├── cases.json # 黄金测试集 │ │ └── run.ps1 # 定期执行脚本 │ ├── escalation/ # 提报规则 │ │ └── routes.json │ ├── wire/ # 外部通道配置 │ ├── handbook/ # 项目知识库 │ │ ├── docs/ │ │ └── prompts/ │ ├── data/ # 持久化数据 + runtime events │ └── _runtime/ # 临时文件（不进 git） │ ├── lab/ # 孵化期：无 git，保留日期前缀 │ └── YYYY-MM-DD-项目名/ # 日期是孵化阶段的临时标记 │ ├── _archive/ # 已完成/归档项目 │ ├── _output/ # 产出物（文章/图片/视频）——不是项目 │ └── 品牌名/ # 每个内容品牌一个子目录 │ └── _templates/ # 项目模板

工具源码（第三方独立工具）

<tools>/ ├── 工具A/ # 第三方工具源码 ├── 工具B/ # ...
第三方工具 ≠ 项目。不进 workspace
统一存放第三方独立工具源码。每个工具一个子目录，~/.agentboard/tools/{id}/manifest.json 的 projectPath 指向这里。
工具注册表（manifest）和工具源码（repo）分离——注册表在 ~/.agentboard/tools/，源码在 <tools>/。

4.2 接口契约

唯一硬接口：notebook/health.json
这是项目对面板暴露的唯一接口。其他所有文件（alarm/、inspector/、escalation/）都是项目内部实现细节。
面板不关心项目怎么跑、用什么跑，只关心：你健康吗？

meta.projectName 是项目的唯一标识。crontab.txt、memory 指针、文档引用全部用这个名字，不用路径。
notebook/health.json — 标准 schema { "status": "ok" | "degraded" | "down", "lastCheck": "2026-06-17T08:00:00+08:00", "score": 95, // 0-100，inspector 打分 "tasks": [ { "name": "task-name", "lastRun": "2026-06-17T08:00:00+08:00", "lastStatus": "success" | "error" | "no_output", "lastDurationMs": 720000, "lastTokens": 45000 } ], "meta": { "projectName": "my-project", "loopPhase": 3, // 0=仅文件, 1=可被扫描, 2=自动化, 3=自愈 "lastAudit": "2026-06-15T..." } }

双界面：内部文件，对外 MCP

执行路径和观测路径不共用传输层
组件内部通信走文件（留言模式，不需要双方同时在线）。对外暴露走 MCP/HTTP API（对话模式，标准协议让任何外部 client 发现和调用）。

内部 · 文件

  • 调度器 → crontab.txt → executor
  • executor → state.json → 面板
  • 项目 → health.json → 面板
  • 写入方不依赖读取方在线
  • Kill 面板，执行照跑
⇔

对外 · MCP / HTTP

  • 面板 → MCP tools/list
  • 面板 → MCP tools/call
  • 面板 → REST /api/cron/state
  • 标准协议，任何 client 可接入
  • 面板崩了，外部看不到但内部不停
为什么不用 MCP 做内部通信：MCP = JSON-RPC over stdio/HTTP，要求 server 活着才能响应。
文件 = JSON on disk，写入方写完就走，读取方什么时候来读都行。单机上留言比对话可靠。

4.3.1 AI 操作说明书 · 触发规则

AI 不靠记忆，靠说明书
传统做法：把规则写在每个 AI 的 system prompt 里 → 多 Agent 间必然漂移。
本架构做法：规则在文件里，AI 自己读。workspace CLAUDE.md 声明每种操作的触发规则，AI 每次操作前先查对应章节。

4.3.2 crontab.txt · executor 合约

调度器不挑 executor
调度器是计时器，不是调度中心。它不绑定任何 AI Agent——crontab.txt 每行声明用哪个 executor。
alarm/crontab.txt # cron_expr executor project-name task [extra...] 0 8 * * * claude-code my-project daily-task 0 9 * * * openclaw-agent content-brand 发布 */30 * * * python screenshot-tool capture --monitor=1
Executor 最小合约
任何 executor 只要满足两条就能接入：
1. 接受 run <project_dir> <task_name> [extra...]
2. stdout 末行输出 JSON：{"status":"ok"|"error","tokens":12345,"durationMs":720000}

换 Agent 不需要改调度器。

4.4 外部 API 调用模型

针对非本地 LLM 的外部 API 调用（生图、语音、第三方数据）。

三个护栏

#护栏机制参考
