0.1 一手理论

📖 Loop Engineering
Addy Osmani · 2026-06-07
Loop Engineering 的原始定义和六原语。
六原语：Automations · Worktrees · Skills · Connectors · Sub-agents · State/Memory
人从「写 prompt 的人」变成「设计系统的人」——人的角色从操作员升级为系统架构师。
关键警告：comprehension debt（理解债）——loop 越快，你理解的和实际存在的之间的差距越大。
cognitive surrender（认知投降）——loop 自己跑起来了，人最容易放弃思考。
同一套 loop 两个人用出相反效果：一个加速自己深度理解的领域，一个逃避理解。

1. 需求清单

循环工具体系要完成的事情，逐项可验证。

  • 闹钟 —— 按 crontab 定时触发 Claude Code 执行任务
  • 巡检 —— 定期扫描项目目录，读取 health.json 汇总状态
  • 提报 —— 异常通过飞书 / 邮件通知，正常不打搅
  • 面板 —— 可视化查看所有项目的健康分和最新运行结果
  • 审计 —— 检查任一项目是否符合循环化标准（Phase 0→1→2→3）
  • 自愈 —— 基础异常自动修复（重启服务、清理临时文件等）
  • 项目注册 —— 新项目一键接入循环体系，不需要写代码
  • 独立运行 —— kill 掉工具架，每个项目仍能独立完成任务
  • 通道扩展 —— 飞书 / 钉钉 / 企业微信等通知通道可插拔
  • 日志追踪 —— 每次触发、执行、结果的完整链路可查
  • 决策记录 —— 架构决策可追溯，不会因对话消失而丢失
  • 文件夹标准化 —— 每个目录放什么有明确约定，新项目照抄即可

2. 联邦制两种形态

集中式已排除。以下聚焦联邦制内部两种实现——Addy Osmani 五组件+记事本，分别归属工具架还是项目。

2.1 五组件归属拆解

每个组件分「定义」和「执行」两层。定义=规则写在哪，执行=谁实际跑。拆开才能判断。

Addy 组件本质定义（规则写哪）执行（谁跑）安全敏感
🔔 闹钟定时触发项目 alarm/crontab.txtagentboard scheduler.js否
🏗 工位Git worktree 隔离Claude Code 原生能力。不属任何一方。—
📖 员工手册项目知识项目 CLAUDE.md + handbook/Claude Code 读取否
📞 电话线外部工具连接项目 wire/（用哪个通道）agentboard MCP + token是。token 不进项目
🔍 质检员独立验证项目 inspector/checks.jsagentboard 调度另一模型否
📝 记事本跨会话状态项目 notebook/health.jsonagentboard 聚合读取否

2.2 两种联邦变体

变体 A · 厚工具架

定义+执行都在 agentboard。项目只是被触发对象。

组件位置：

  • 闹钟定义 → agentboard cron/tasks.json
  • 员工手册 → agentboard skills/
  • 电话线 → agentboard 全管
  • 质检规则 → agentboard rules/
  • 记事本 → agentboard state/

一句话：工具架是大脑，项目是手脚。

变体 B · 薄工具架

定义在项目，执行在工具架。工具架只做调度+聚合。

组件位置：

  • 闹钟定义 → 项目 alarm/crontab.txt
  • 员工手册 → 项目 CLAUDE.md + handbook/
  • 电话线 → token 在 agentboard，规则在项目 wire/
  • 质检规则 → 项目 inspector/checks.js
  • 记事本 → 项目 notebook/health.json

一句话：工具架是代理，项目是主人。

2.3 多维度对比

维度变体 A · 厚工具架变体 B · 薄工具架
解耦agentboard 熄了，定时任务+质检+通知全停agentboard 熄了，项目用 Windows 任务计划+原生 cron 继续跑。只丢 UI
安全token 集中在 agentboard ✅token 同样集中。项目 wire/ 只写「用飞书」，不含密钥 ✅
新项目接入在 agentboard 手动加 3 处配置项目内建 2 个文件，agentboard 零改动自动发现
项目迁移搬项目=agentboard 配置也要搬复制文件夹即迁移。新 agentboard 自动发现
调试闹钟没触发？查 agentboard 日志先查项目 data/ 日志，再查 agentboard。责任链清晰
多项目差异化不同质检规则=agentboard 里写 if-else每个项目自带 inspector/checks.js，互不干扰
工具架替换换工具架=所有配置迁移换工具架=重新指向项目路径
版本控制agentboard 配置进不进 git 都尴尬项目配置随项目进 git
结论初期快，但每加一个项目 agentboard 就胖一圈前期多写几个文件，项目越多优势越明显

2.4 Karpathy 验证：成功标准直接判

四条硬标准，打钩说话
1. kill 掉 agentboard，evopearl-data 能不能独立跑完今天的日报？
  变体 A：✗（闹钟和记事本都在 agentboard，停了就没了）
  变体 B：✓（crontab.txt + health.json 在项目，Windows 任务计划可兜底）

2. 把 evopearl-data 复制到新机器，能不能直接工作？
  变体 A：✗（需手动重建 agentboard tasks/rules/state）
  变体 B：✓（项目自包含，agentboard 自动发现）

3. 飞书 token 泄露风险哪个更大？
  平手。两变体 token 都在 agentboard，不进项目仓库。

4. 新加一个项目要改多少地方？
  变体 A：agentboard 改 3 处（tasks.json + rules + state）
  变体 B：项目内新建 2 文件（CLAUDE.md + notebook/health.json），agentboard 零改动

得分：A = 1/4，B = 4/4。变体 B 胜出。

3.3 任意新项目的标准骨架

项目根目录\ ├── CLAUDE.md # 项目规则 └── notebook/ # 标准接口（agentboard 读取此层） └── health.json # 必须：{ "status":"ok", "lastCheck":"...", "score":95 }

一个项目只要有 notebook/health.json，agentboard 就能自动发现并展示在面板上。loop-audit 技能也读同一个文件。这是唯一的硬性接口约定。

Addy 组件决定理由
🔔 闹钟定义在项目 alarm/crontab.txt，执行在 agentboard scheduler调度需要常驻进程；但规则属于项目，换机器也能跑

6. SOP · 新项目接入循环体系

1

创建项目骨架

在项目根目录建 CLAUDE.md + notebook/health.json。最少两个文件即可。

2

写 health.json

按标准格式填写：{"status":"ok","lastCheck":"2026-06-15T...","score":95,"tasks":[]}

3

运行 loop-audit（可选但建议）

在项目目录执行 loop-audit，检查是否满足循环化标准。Phase 0 门禁：文件存在 + 格式正确。

4

agentboard 自动发现

agentboard 启动时扫描项目目录，读到 health.json 自动展示在面板上。无需手动注册。

5

（可选）加闹钟 / 巡检 / 提报

按需在 alarm/、inspector/、escalation/ 下放配置文件。不加也能跑，加了自动化更强。
