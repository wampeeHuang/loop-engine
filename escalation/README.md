# 提报 — 分级推送

异常需要人决策时，系统主动找人，不是人等系统。**刹车——不是为了通知，是为了等人批准。**

## 四级模型

| 级别 | 触发条件 | 动作 | 示例 |
|------|---------|------|------|
| L0 自动处理 | 首次失败 | 重试 ≤ 3 次，不提报 | HTTP 超时 → 60s 后重试 |
| L1 汇总 | 当天所有任务完成 | 发日报摘要 | "今日 3/3 完成，产出 12 条" |
| L2 提报 | 连续 2 天同一 job 失败 / 数据量降 50%+ | 推送消息，等人确认 | 飞书卡片通知 |
| L3 阻断 | 账单欠费 / API key 失效 / 部署不回滚 | 立即推送 + 停止当天所有任务 | 飞书 + 红色告警 |

## 规则定义

`escalation/rules.json` — 声明式规则，机器可解析，人类可读。

## 通知渠道

通过 `wire/notify.ps1` 推送。webhook URL 从环境变量 `FEISHU_WEBHOOK` 读取。

## 设计原则

1. **L2 以上才找人**：L0-L1 自动处理，不骚扰人。人麻木了比没通知更危险
2. **通知说清楚要做什么**："健康分从 2.5 降到 1.8"不够，"inspector 组件从 ok 降到 missing，检查 inspector/verify.js 是否被删除"
3. **有超时机制**：L2 提报后 24 小时无人确认 → 升级为 L3
4. **提报不是监控**：提报频道 ≠ 日志频道。不要每跑一次发一条"运行成功"
