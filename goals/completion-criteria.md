## loop-engine 自身的完成标准

### 战略目标
1. 每个被审计项目有 `CLAUDE.md` + `notebook/health.json`
2. 联邦面板能显示所有项目的真实健康状态
3. loop-audit 技能保持自迭代

### 战术目标

| 目标 | 验证方法 | 通过条件 | 失败动作 |
|------|---------|---------|---------|
| 每周自检 | inspector/verify.js | exit_code=0，无 issues | 标记异常，events.jsonl 记录 |
| 交叉验证 | 随机抽查 1 个项目 | health 和产出一致 | L2 提报 |
| 面板数据新鲜 | 检查 agentboard /api/loop/health | 返回 200，数据 ≥ 1 条 | L3 告警 |
| 审计报告不过期 | 检查 LOOP_AUDIT.md mtime | ≤ 30 天 | 下次闹钟触发重审 |

### 边界约束

- 面板纯读层——不通过 API 控制任何项目
- 审计不自动修改被审计项目的文件——只输出建议
- 不引入中央监控引擎——各项目自包含
- health.json 的 today.jobs_completed 必须 ≤ jobs_total

### 降级路径

```
单次失败 → 重试 (≤3) → 标记"今天失败" → 明天自动重试 → 连续 3 天 → L3 提报
面板不可用 → 各项目独立运行不受影响 → L3 告警 → 修面板
审计框架过时 → loop-audit 自迭代机制触发 → 提示用户审阅框架修改
```
